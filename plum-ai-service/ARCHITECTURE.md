ARCHITECTURE.md
Техническая карта проекта plum-ai-agent (Plum Dev) по состоянию текущей кодовой базы. Этот документ нужен как опорный файл для будущих изменений: перед правками в пайплайне, роутинге, Supabase-состоянии или промптах сначала сверяться с ним.

1. ОБЩАЯ СХЕМА ПАЙПЛАЙНА (High-Level Flow)
Точка входа приложения
app/main.py

create_app() создает FastAPI-приложение Plum Dev AI Service.

В app.state кладутся:

settings = get_settings() из app/config.py.

gemini = GeminiService(settings) из app/gemini_service.py.

supabase = SupabaseService(settings) из app/supabase_service.py.

Подключается api.router.

/health возвращает {"status": "ok"}.

Основной API endpoint
app/api.py

Роут: POST /api/v1/chat.

Схема входа: ChatRequest из app/schemas.py:

channel: telegram, whatsapp, instagram.

chat_id.

instance_id.

message.

chat_history.

reset_context.

Схема выхода: ChatResponse:

route.

routes.

answer.

checkout.

metadata.

Полный путь входящего сообщения
Внешний канал отправляет сообщение в POST /api/v1/chat.

api.chat() получает из request.app.state готовые сервисы:

GeminiService.

SupabaseService.

_check_rate_limit(channel, chat_id) ограничивает частоту сообщений.

supabase.get_tenant_settings(instance_id=payload.instance_id) загружает tenant-настройки (настройки бота Plum Dev).

Если reset_context=True, supabase.clear_conversation_state() удаляет состояние этого диалога.

supabase.get_last_message_at() берет последний created_at из логов.

_is_new_session(last_message_at) проверяет, просрочена ли сессия (более 6 часов).

Если сессия не новая, поднимается история через supabase.fetch_recent_chat_history().

_build_effective_history() объединяет серверную и клиентскую историю.

_strip_generation_fallback_history() удаляет системные fallback-фразы, чтобы не зациклить модель.

supabase.get_chat_session_metadata() читает метаданные диалога.

_merge_client_facts_for_request() извлекает B2B-факты через gemini.extract_client_facts():

ниша бизнеса.

текущий объем лидов/заявок.

наличие CRM.

главный запрос (бот, авто-корзина, поддержка).

Найденные факты сохраняются в chat_sessions.metadata.client_facts.

До LLM-роутера выполняется детерминированный перехват запросов портфолио:

_is_document_request(payload.message).

Если клиент просит кейсы, примеры работ, портфолио, API сразу возвращает ссылку на кейсы Plum Dev (без вызова RAG).

Поднимается долгосрочная память (если диалог длинный).

gemini.classify_routes() выбирает маршруты Route.

_has_explicit_commercial_intent() проверяет, есть ли намерение купить/заказать аудит.

Фильтрация CHECKOUT: если коммерческого намерения нет, роут удаляется.

_select_primary_route() выбирает приоритетный маршрут (RAG_REQUIRED -> CHECKOUT).

Если нужен RAG_REQUIRED:

Поиск по векторной базе Supabase (услуги Plum Dev, интеграции, технические возможности).

Формирование rag_context.

Если нужен CHECKOUT:

Подключается прайс/коммерческий контекст.

Создается CheckoutPayload для записи на звонок или брифинг.

_answer_with_rag_retry() вызывает Gemini для генерации ответа.

Формируется ChatResponse.

Ответ и метаданные пишутся в chat_logs.

Генерация ответа внутри GeminiService
Последовательность шагов:

Сбор промпта: сообщение, история, B2B-факты клиента, контекст памяти, RAG (база знаний Plum Dev), коммерческий контекст.

В системные инструкции подгружаются SALES_MASTER_PROMPT, защитные гвардейлы и UNIFIED_COMMERCIAL_RULES_PROMPT (адаптированные под продажу ИИ-услуг).

_ensure_critical_commercial_trigger_rule() проверяет наличие правил продаж.

Выполняется вызов модели _generate_text().

Пост-обработка _avoid_repeated_answer() и _avoid_repeated_closing_phrase() заменяет заезженные вопросы (например, запрет на частое использование "Что скажете?").

2. МАРШРУТИЗАЦИЯ И РОУТЕР (Routing Logic)
Route enum
Допустимые маршруты строго фиксированы:

GENERAL

RAG_REQUIRED

CHECKOUT

Роутер как купол системы
Запрещено обходить роутер. Он решает, нужно ли вызывать базу знаний или переводить клиента на этап оформления/записи.

Route: GENERAL
Приветствия, small talk, нерелевантные вопросы. RAG не вызывается.

Route: RAG_REQUIRED
Назначение: вопросы об архитектуре ИИ, сроках разработки, возможностях интеграции (Битрикс24, amoCRM, Telegram, Instagram, вебсайт), кейсах, технических ограничениях.

Ключевые слова-эвристики: интеграция, crm, модели, chatgpt, llm, сроки, база знаний, безопасность данных, api, как работает.

Route: CHECKOUT
Назначение: клиент хочет узнать точную цену, заказать разработку, записаться на аудит или созвон.

Ключевые слова: цена, стоимость, сколько стоит, начать, оплатить, заказать, созвон, бриф, аудит.

Критическая защита: проверяется через _has_explicit_commercial_intent(). Бот не переходит к брифу от простого вопроса "как это работает".

Детерминированный document-request gate (Перехват портфолио)
Паттерны: кейсы, портфолио, примеры работ, покажи что делали, реализованные проекты.

Действие: API обходит LLM и выдает захардкоженный ответ со ссылкой на портфолио Plum Dev, возвращая пользователя к обсуждению его ниши.

3. БАЗА ДАННЫХ И СТЕЙТЫ (Database & State Management)
Архитектура Supabase (chat_logs, chat_sessions, user_memories, tenants) остается неизменной.

Client facts / chat_sessions
В metadata.client_facts теперь сохраняются бизнес-показатели:

business_niche (ниша компании).

lead_volume (объем заявок).

crm_system (используемая CRM).

target_solution (какую автоматизацию хочет клиент).

Knowledge base / RAG
Таблица public.knowledge_base содержит чанки с описанием услуг Plum Dev, техническими возможностями ИИ-агентов, стандартами безопасности и регламентами работы.

4. СИСТЕМА ОБРАБОТКИ ОШИБОК И ФОЛЛБЭКОВ
Механика try-except и retry-запросов остается прежней.

_build_generation_fallback_answer()
В случае критического сбоя LLM безопасный fallback возвращает универсальный ответ:

"Я специализируюсь на создании кастомных ИИ-агентов и авто-корзин для бизнеса. Давайте обсудим вашу нишу и рассчитаем стоимость внедрения. Напишите, какую задачу вы хотите автоматизировать?"
Fallback не выдумывает цены на разработку и сроки.

5. КРИТИЧЕСКИЕ ОГРАНИЧЕНИЯ (Strict Constraints)
Приоритет результата (ROI и Автоматизация)
Главная задача бота: продавать выгоду для бизнеса (экономию времени, повышение конверсии, работу 24/7), а не просто технический код.

При ответах на вопросы всегда связывать технологию с бизнес-результатом клиента.

Запрет на галлюцинации цен и сроков
Бот озвучивает цены СТРОГО на основе актуального коммерческого прайса из БД (например, тарифы на базовых ботов).

Если проект нестандартный, боту КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО выдумывать смету. Он обязан направить клиента на созвон: "Точная стоимость зависит от вашей CRM и базы данных. Давайте назначим короткий созвон для оценки".

Матрица отказов
Технический отказ: Клиент не хочет читать кейсы или смотреть детали интеграции. Бот фиксирует это и возвращается к сбору бизнес-потребностей.

Коммерческий отказ: Клиент говорит "дорого" или "пока не готов". Включается отработка возражений: акцент на том, сколько денег бизнес теряет без ИИ прямо сейчас, и предложение начать с дешевого тест-драйва (аудита).

Anti-repetition
Запрет на использование одинаковых закрывающих вопросов. Использовать вариации: "Рассказать, как это будет работать у вас?", "Готовы обсудить интеграцию?", "Сделаем бесплатный аудит вашей воронки?".

Checkout (Закрытие в лид)
Если клиент готов двигаться дальше, бот перестает консультировать и собирает данные:

Имя.

Ниша бизнеса.

Удобное время для созвона (или номер для Telegram/WhatsApp).
Альтернативный вариант Checkout: выдача Inline-кнопки на Calendly или форму заявки.

6. ФАЙЛЫ, КОТОРЫЕ НЕЛЬЗЯ ПРАВИТЬ ВСЛЕПУЮ
app/api.py: роутинг, сбор истории, перехват кейсов.

app/gemini_service.py: пайплайны генерации, парсинг роутов, извлечение B2B-фактов.

app/supabase_service.py: логика сохранения контекста и логов.

SQL-файлы: структура векторной базы.

7. ИНВАРИАНТЫ ДЛЯ БУДУЩИХ ИЗМЕНЕНИЙ
Маршрутный контракт (GENERAL, RAG_REQUIRED, CHECKOUT) нерушим.

Логика извлечения фактов (ниша, лиды) должна работать на каждом сообщении.

Любое изменение fallback не должно возвращать пользователю системные логи или ошибки.

При сбоях генерации вопросы должны браться из безопасного резерва, не прерывая продажу услуг Plum Dev.
